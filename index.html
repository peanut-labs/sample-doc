Sample API

Overview

Peanut Labs Sample API, a REST based API, can be programmatically used to connect your client surveys directly with Peanut Labs panelists. This API document describes the usage and resources needed for integrating with Sample API.

API Structure

Sample API is organized around REST. Below table describes the Sample API URI structure:

  • URI

    {Base URI}/{Version}/{Resources}

  • Base URL

    https://sample.peanutlabs.com/sample (Production)
    https://api-dev.peanutlabs.com/sample (Test)

    Sample API supports API call over HTTP, however we strongly recommend you to use HTTPS.

  • /Version

    v1

  • /Resources

    • /projects [GET | POST]
    • /quotas [GET | POST]
    • /feasibility [POST]

  • Response.ContentType

    application/JSON

Getting Started

Authentication & Authorization

You authenticate all your requests with Sample API server by sending an HTTP authorization header as specified below:

advertiserâ€™sId:timestamp:HMAC

Sample API server verifies the HTTP Authorization Header before allowing access to Peanut Labs system.

Here are all the parameters you will need to generate an HTTP Authorization Header.

  • Advertiserâ€™s Id

    A unique identifier provided to you by Peanut Labs.

  • secret_key

    A security key provided to you by Peanut Labs.

    Please do not expose your key with anyone or in public. This could compromise the integrity of your data.

  • timestamp

    Timestamp when the request was generated.

  • HMAC

    Encrypted SHA256 hash to verify your request.
    HMAC = SHA256 of (request_method + request_uri + all API parameters in alphabetical order + timestamp) using your secret_key

     

    For the API parameters:

    • sort the argument using the first-level hash
    • stringify the values even if the value is a complex object
    • concat the values together in alphabetical order

Response & Error codes

Sample API uses standard HTTP status code to indicate success or failure of an API request. In general 2xx range would indicate success, 4xx range would indicate an error (such as missing parameters, invalid field or authorization failure) and 5xx will indicate an issues with Sample API server.

Sample API response will also return an application/JSON object with more information about the API operation - a resultCode to indicate success or failure of an API operation; an optional resultError to indicate a detailed error description, if any; and an optional data field to return data from the API operation.

  • resultCode
    0 or 1

    1 for success; 0 for error.

  • resultError
    string, optional only present when resultCode = 0

    A detailed error message describing what went wrong.

  • <data>
    optional

    A placeholder to return the result of an API operation.

HTTP Status code summary

  • 200

    Everything worked as expected

  • 400

    Bad request

  • 403

    Invalid security hash

  • 404

    Resources not found

  • 500

    Something went wrong on Peanut Labs end

 

Example 1: Success response

HTTP Status Code: 200

{
  "resultCode": 0
}

 

Example 2: Error response

HTTP Code: 400

{
  "resultCode": 1, 
  "resultError": "Project doesn't exist"
}

API Methods

Setup a Project

POST /projects

Creating a project is the first step to launching your surveys to Peanut Labs panelist. This API is used to create a project.

Request

  • extProjectId
    required string

    A unique identifier for your project.

  • title
    string

    A project title of your choosing.

  • link
    Required string

    Survey URL where Peanut Labs would send its panelist to take the survey.


    Link should contain a placeholder #sid# which Peanut Labs will replace with an unique session identifier trackerId to track respondents.

  • country
    REQUIRED string

    2 letter ISO country codes.

  • currency
    string

    USD is the only currency supported.

  • notificationEmail
    string

    The email address to receive notifications, if project was paused due to conversion rules.

  • indicativeIncidence
    REQUIRED 0 <= integer <= 100

    Percent of respondent you expect to qualify for the survey.

    Your survey project may be paused if actual conversion rate is lower than indicative incidence by 25% or more. Notification email will be sent to notificationEmail when such happens.

  • lengthOfInterview
    required integer >= 0; in mins

    Average number of minutes it takes an average user to complete your survey.

  • daysInField
    REQUIRED integer >= 0; in days

    Number of days the survey should be active in the field.

    We donâ€™t actively monitor daysInField but there are scenarios when long running projects beyond daysInField are paused.

  • deviceTargeting
    hash

    Placeholder for deviceType list.

    • Show child attributes

    • deviceType
      array of string; defaults to all devices

      Devices our respondents will take the survey on. Supported devices are desktop, mobile, and tablet.

      You don't need to provide this if your survey is available on all devices.

  • inclusions
    hash

    Inclusions will let you expose traffic to respondents who were part of your previous projects.
    See inclusion/exclusion for an example.

    • Show child attributes

    • completes
      array of extProjectId

      Includes respondents who had completed your previous projects.

    • respondents
      array of extProjectId

      Includes respondents who had attempted your previous projects.

    • dropouts
      array of extProjectId

      Includes respondents who were dropped out of your projects midway.

    • overquota
      array of extProjectId

      Includes respondents who got disqualified from previous projects because of overquota.

    • screenout
      array of extProjectId

      Includes respondents who were screened out of previous projects.

  • exlusions
    hash

    Like inclusions, exclusions can be used to NOT expose traffic to respondents who were part of your previous projects.
    See inclusion/exclusion for examples.

    • Show child attributes

    • completes
      array of extProjectId

      Excludes respondents who had completed your previous projects.

    • respondents
      array of extProjectId

      Excludes respondents who had attempted your previous projects.

    • dropouts
      array of extProjectId

      Excludes respondents who were dropped out of your projects midway.

    • overquota
      array of extProjectId

      Excludes respondents who got disqualified from previous projects because of overquota.

    • screenout
      array of extProjectId

      Excludes respondents who were screened out of previous projects.

  • preScreener
    hash prescreener; defaults to Peanut Labs fraud prescreener

    A set of screening question that users take before qualifying for a survey. See precreener example on how to define prescreeners.

    Peanut Labs fraud prescreener will be used for no/empty prescreener definition.

  • autoLaunch
    0 or 1; 0 to launch in test mode, 1 to launch to live traffic; defaults to 0

    This controls project launch. Setting this to 1 will launch the project as soon as it is created. 0 would only launch the project in test mode allowing you to test all the redirect links.

    We highly recommend you test your projects for all the redirect links before taking it live.

  • quotaSetting
    required array of hash

    Defines selection criteria for panelists according to attributes such as age, sex, etc.

    • Show child attributes

    • extQuotaId
      required string

      A unique quota identifier across all your projects.

    • completes
      required integer >= 0

      Total number of completes required for this quota.

    • cpi
      required decimal >= 0.00

      Amount Peanut Labs will be paid for each completes in USD for this quota.

      USD is the only currency supported.

    • attributes
      required hash; each key in the hash is the attributeId of that attribute

      Targeting attributes for panelists within this quota. Total allocation within each sub-attributes should sum up to 100. See attributes and nesting for examples.

      • Show child attributes

      • option
        string

        attributeId option from the list of available targeting attributes.

      • perc
        0 <= integer <= 100

        Percentage of respondents desired for this attributeId.

    • filters
      hash of array; each key in the hash is the attributeId of that attribute

      Filters are just like attributes which all panelists should belong to before qualifying for a survey. See filters for examples.

      Zip code can only be used as filters. An API call cannot have more than 10,000 zip codes in the filter list.

    • dailyCap
      integer >= 0; 0 means no limit

    • nested
      array

      Nesting will apply an inner level targeting within an already targeted group.

      See attributes and nesting for examples.

      Nesting can run upto three level deep.

 

Response

  • callbacks
    hash

    Callback URL where you should redirect respondents after the survey. Each callback URL contains two placeholders - #sid# and #hash# which should be replaced before you call the callback URLs.

    Replace #sid# by trackerId which was passed to you when Peanut Labs redirected user to your survey.

    Replace #hash# with client-side generate hash.
    #hash# = MD5 of (status + trackerId + shared secret_key)

    See secure exit redirects for an example

    • Show child attributes

    • Complete
      string, URL encoded

      Redirect URL where you should send respondents to after they complete the survey successfully.

    • OverQuotas
      string, URL encoded

      Redirect URL where you should send respondents to if quota was already filled.

    • ScreenOut
      string, URL encoded

      Redirect URL where you should send respondents to if they disqualified from the survey based on screening questions.

  • resultCode
    0 or 1

    1 for success, 0 for error

  • resultError
    string,
    only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Update Project Status

POST /projects/:{extProjectId}/{pause | resume | launch | close}

This method updates a project status with specified action.

Action

  • pause

    Stop traffic to all quotas for this project.

  • resume

    Resume traffic to all quotas for this project.

  • launch

    Takes a project live for the first time. This is only used when project was created with autoLaunch set as 0.

  • close

    Stops all traffic flow into the project and closes it. Once a project is closed it cannot be reopened.

    Only use this when you have determined your project is complete and can be closed.

 

Response

  • resultCode
    0 or 1

    1 for success; 0 for error.

  • resultError
    string, only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Definition

POST https://api-dev.peanutlabs.com/rest/v1/projects/project001/pause

 

Example Response
{
  "resultCode": 1
}

Targeting Attributes and Nesting

Targeting allows you to obtain different number of completes from different types of end users by associating a percentage of the required completes desired with an attribute value.

The most simplest form of targeting is to just provide the targeting attributes without any nesting structure. With this targeting, you can specify a requirement like:

  • 100% of respondents to be male or female, and
  • 100% of respondents to be within the age group of 18-25 or 26-45
  • without caring to filter within within each attributes.

In this above scenario it is likely that all males could be of the age group 18-25 and all females within the age group of 26-45, or any combination between the two.

If you desire specific control over the mix of attributes within a composition then you may choose to define nested attributes. In the above scenario with nesting you can achieve a mix to be like:

  • 25% male within the age group of 18-25,
  • 25% male within the age group of 26-45,
  • 25% female within the age group of 18-25, and
  • 25% female within the age group of 26-45.
Nesting allows you to have a fine grain control over the mix of attributes.
Example without nesting
{
  "attributes": {
    "sex": [
      {
        "option": "1", 
        "perc": "50"
      },
      {
        "option": "2", 
        "perc": "50"
      }
    ],
    "age": [
      {
        "option": "18-25", 
        "perc": "20"
      },
      {
        "option": "26-45", 
        "perc": "80"
      }
    ]
  }
}

The above example will target the following panel composition:

  • 50% of the respondents will be male and the other 50% female.
  • 20% of the respondents will be within the age group 18-25 and the other 80% within the age 26-45.
  • Since age group is not nesting with sex, it is possible that you may get a panel composition as:
    • 20% male within the age of 18-25,
    • 30% male within the age of 26-45, and
    • 50% female within the age of 26-45.

 

Example with nesting
{
  "attributes": {
    "sex": [
      {
        "option": "1", 
        "perc": "50"
      },
      {
        "option": "2", 
        "perc": "50"
      }
    ],
    "age": [
      {
        "option": "18-25", 
        "perc": "50"
      },
      {
        "option": "26-45", 
        "perc": "50"
      }
    ],
    "q121": [
      {
        "option": "qx121-2", 
        "perc": "50"
      },
      {
        "option": "qx121-1", 
        "perc": "50"
      }
    ]
  },
  "nested": [
    [
      "age", 
      "sex"
    ],
    [
      "q121"
    ]
  ]
}

The above example will targets the following panel composition:

  • 25% male within age 18-25
  • 25% male within age 26-45
  • 25% female within age 18-25
  • 25% female within age 26-45
  • Of the above composition 50% will be single and 50% will be married

Filters

Filters are attributes where all users should have before qualifying for the survey.

  • Zip code can only be used as filters
  • An API call cannot have more than 10,000 zip codes in the filter list
{
  "q101": [
    "qx101-1", 
    "qx101-2", 
    "qx101-3"
  ]
}

Respondents without attribute qx101-1, qx101-2, qx101-3 will not be invited for the survery.

Secure Exit Redirects

After the user completes the survey, user should be redirected to a secure exit redirect links. The link contains an encrypted hash which Peanut Labs validates before updating the user status for the session.

The hash parameter can be generated as follows:
#h# = MD5(status + trackerId + secret_key)

where,
status = C for completes, P for screenout and F for overquota;
trackerId = Peanut Labs passed userâ€™s session identifier along with the survey entry link;
secret_key = A security key provided to you by Peanut Labs.

Please do not expose your key with anyone or in public. This could compromise the integrity of your data.
Example to compute hash

To compute hash, letâ€™s say you have:

  • status = C
  • trackerId = 123a456
  • secret_key = The dog jumped over the house

then,

#h# = md5(C123a456The dog jumped over the house)

#h# = 7a0ee4af327cfee9854d963e2a2b1fbf

The resulting redirect URL becomes:

https:\/\/api-callbacks.peanutlabs.com\/transactionComplete.php?trackerId=123a456&status=C&h=7a0ee4af327cfee9854d963e2a2b1fbf
