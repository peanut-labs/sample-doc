Samplify API Reference

Samplify API, a REST based API, can be programmatically used to connect your client surveys directly with Research Now consumer panel. This API document describes the usage and resources needed for integrating with Samplify API.

Our API has predictable, resource-oriented URLs, and uses HTTP response codes to indicate API errors. JSON is returned by all API responses, including errors. To make the APIs explorable, we provide a Test account and a Live account. Requests sent to the test account does not go to our actual panelists and therefore does not incur any cost.

We also provide a PHP SDK that you can use directly to make API requests. Please checkout this Github page for the SDK. It provides built in logic for Authentication, and therefore makes it easier to consume our APIs.

Production API endpoint
https://sample.peanutlabs.com/sample/v1/{METHOD}
Sandbox API endpoint
http://api-dev.peanutlabs.com/sample/v1/{METHOD}

We support API calls over HTTP, however we strongly recommend you to use HTTPS.

Key Concepts

Terminologies

Here are all some of the key terminologies that is used extensively in this documentation, and its therefore important to understand what they mean

  • Project

    A Project is the top most level resource that represents a single survey.

  • Quota

    A Quota allows you to target survey takers based on certain demographics, and is also the numerical representation of the required completes. Defining a quota is mandatory when setting up a project.

  • Feasibility

    Feasibility is the analysis and evaluation of a study, to determine how much completes is possible from a certain demographic.

State Management

Every Project/Quota in the Research Now system has an associated state that determines its behavior and the operations that are allowed within that state. It is therefore important to understand the different states and how transitions work within each state.

Please see the state diagram on the right to learn about the lifecycle of Projects/Quotas and how they transisiton from one state to another

Here is a list of valid states of project/quotas and what the state means

  • PreLaunch

    Project/Quota is created in our system, but is not open to our users to take it.

  • Launched

    Project/Quota is live and can get traffic.

  • Paused

    Project/Quota is temporarily cut off from getting traffic.

  • Closed

    Project/Quota is permanently closed and cannot be reopened.


Key Things to Remember about states:

  • A Quota can be in launched state only if the project is launched
  • Pausing a Project, pauses all the launched quotas
  • Launching a paused Project, launches all paused quotas
  • Quota/Project can move to Paused/Closed states by our monitoring system, and they cannot be relaunched again

Getting Started

Authentication

You authenticate all your requests with Samplify API server by sending an HTTP authorization header as specified below:

advertiser’sId:timestamp:HMAC

Samplify API server verifies the HTTP Authorization Header before allowing access to Research Now system.

Here are all the parameters you will need to generate an HTTP Authorization Header.

  • Advertiser’s Id

    A unique identifier provided to you by Research Now

  • secret_key

    A security key provided to you by Research Now.

    Please do not expose your key with anyone or in public. This could compromise the integrity of your data.

  • timestamp

    Timestamp when the request was generated.

  • HMAC

    Encrypted SHA256 hash to verify your request.
    HMAC = SHA256 of (request_method + request_uri + all API parameters in alphabetical order + timestamp) using your secret_key

     

    For the API parameters:

    • sort the argument using the first-level hash
    • stringify the values even if the value is a complex object
    • concat the values together in alphabetical order

Example

For setup a project (example from below), your JSON body may look like:

{"extProjectId": "project001", "title": "Test Survey", "link": "www.mysurvey.com?uid=#sid#", "country": "US", "currency": "USD", "notificationEmail": "testpm@abc.com", "indicativeIncidence": 20, "lengthOfInterview": 16, "daysInField": 5, "deviceTargeting": {"deviceType": [ "mobile"]}, "inclusions": [{"completes": ["project002"], "screenouts": ["project002"]}], "autoLaunch": 1, "quotaSetting": [{"extQuotaId": "invite001", "completes": "5", "cpi": "3", "attributes": {"q101": [{"option": "qx101-6", "perc": "50"}, {"option": "qx101-5", "perc": "50"}]}, "filters": {"q100": ["qx100-2"]}, "nested": [["q101"]]}, {"extQuotaId": "invite002", "completes": "10", "cpi": "5", "attributes": {"age": [{ "option": "18-30", "perc": "50"}, {"option": "31-49", "perc": "50"}]}, "nested": [["age"]]}]}

Response & Error codes

Samplify API uses standard HTTP status code to indicate success or failure of an API request. In general 2xx range would indicate success, 4xx range would indicate an error (such as missing parameters, invalid field or authorization failure) and 5xx will indicate an issues with Samplify API server.

Samplify API response will also return an application/JSON object with more information about the API operation - a resultCode to indicate success or failure of an API operation; an optional resultError to indicate a detailed error description, if any; and an optional data field to return data from the API operation.

  • resultCode
    0 or 1

    1 for success; 0 for error.

  • resultError
    string, optional only present when resultCode = 0

    A detailed error message describing what went wrong.

  • <data>
    optional

    A placeholder to return the result of an API operation.

HTTP Status code summary

  • 200

    Everything worked as expected

  • 400

    Bad request

  • 403

    Invalid security hash

  • 404

    Resources not found

  • 500

    Something went wrong on Research Now end

 

Example 1: Success response

HTTP Status Code: 200

{
  "resultCode": 1
}

 

Example 2: Error response

HTTP Code: 400

{
  "resultCode": 0, 
  "resultError": "Project doesn't exist"
}

API Methods

Setup a Project

POST /projects

Creating a project is the first step to launching your surveys to Research Now panelist. This API is used to create a project.

Request

  • extProjectId
    required string

    A unique identifier for your project.

  • title
    string

    A project title of your choosing.

  • link
    Required string

    Survey URL where Research Now would send its panelist to take the survey.


    Link should contain a placeholder #sid# which Research Now will replace with a unique session identifier trackerId to track respondents.

  • country
    REQUIRED string

    2 letter ISO country codes.

  • currency
    string

    USD is the only currency supported.

  • notificationEmail
    string

    The email address to receive notifications, if project was paused due to conversion rules.

  • indicativeIncidence
    REQUIRED 0 <= integer <= 100

    Percent of respondent you expect to qualify for the survey.

    Your survey project may be paused if actual conversion rate is lower than indicative incidence by 25% or more. Notification email will be sent to notificationEmail when such happens.

  • lengthOfInterview
    required integer >= 0; in mins

    Average number of minutes it takes an average user to complete your survey.

  • daysInField
    REQUIRED integer >= 0; in days

    Number of days the survey should be active in the field.

    We don’t actively monitor daysInField but there are scenarios when long running projects beyond daysInField are paused.

  • deviceTargeting
    hash

    Placeholder for deviceType list.

    • Show child attributes

    • deviceType
      array of string; defaults to all devices

      Devices our respondents will take the survey on. Supported devices are desktop, mobile, and tablet.

      You don't need to provide this if your survey is available on all devices.

  • inclusions
    hash

    Inclusions will let you expose traffic to respondents who were part of your previous projects.
    See inclusion/exclusion for an example.

    • Show child attributes

    • completes
      array of extProjectId

      Includes respondents who had completed your previous projects.

    • respondents
      array of extProjectId

      Includes respondents who had attempted your previous projects.

    • dropouts
      array of extProjectId

      Includes respondents who were dropped out of your projects midway.

    • overquota
      array of extProjectId

      Includes respondents who got disqualified from previous projects because of overquota.

    • screenout
      array of extProjectId

      Includes respondents who were screened out of previous projects.

  • exclusions
    hash

    Like inclusions, exclusions can be used to NOT expose traffic to respondents who were part of your previous projects.
    See inclusion/exclusion for examples.

    • Show child attributes

    • completes
      array of extProjectId

      Excludes respondents who had completed your previous projects.

    • respondents
      array of extProjectId

      Excludes respondents who had attempted your previous projects.

    • dropouts
      array of extProjectId

      Excludes respondents who were dropped out of your projects midway.

    • overquota
      array of extProjectId

      Excludes respondents who got disqualified from previous projects because of overquota.

    • screenout
      array of extProjectId

      Excludes respondents who were screened out of previous projects.

  • preScreener
    hash prescreener; defaults to Research Now fraud prescreener

    A set of screening question that users take before qualifying for a survey. See precreener example on how to define prescreeners.

    Research Now fraud prescreener will be used for no/empty prescreener definition.

  • autoLaunch
    0 or 1; 0 to launch in test mode, 1 to launch to live traffic; defaults to 0

    This controls project launch. Setting this to 1 will launch the project as soon as it is created. 0 would only launch the project in test mode allowing you to test all the redirect links.

    We highly recommend you test your projects for all the redirect links before taking it live.

  • quotaSetting
    required array of hash

    Defines selection criteria for panelists according to attributes such as age, sex, etc.

    • Show child attributes

    • extQuotaId
      required string

      A unique quota identifier across all your projects.

    • completes
      required integer >= 0

      Total number of completes required for this quota.

    • cpi
      required decimal >= 0.00

      Amount Research Now will be paid for each completes in USD for this quota.

      USD is the only currency supported.

    • attributes
      required hash; each key in the hash is the attributeId of that attribute

      Targeting attributes for panelists within this quota. Total allocation within each sub-attributes should sum up to 100. See attributes and nesting for examples.

      • Show child attributes

      • option
        string

        optionId option from the list of available targeting attribute options.

      • perc
        0 <= integer <= 100

        Percentage of respondents desired for this optionId.

    • filters
      required hash of array; each key in the hash is the attributeId of that attribute

      Filters are just like attributes which all panelists should belong to before qualifying for a survey. See filters for examples.

      Zip code can only be used as filters. An API call cannot have more than 10,000 zip codes in the filter list.

    • dailyCap
      integer >= 0; 0 means no limit

    • nested
      required array

      Nesting will apply an inner level targeting within an already targeted group.

      See attributes and nesting for examples.

      Nesting can run upto three level deep.

 

Response

  • callbacks
    hash

    Callback URL where you should redirect respondents after the survey. Each callback URL contains two placeholders - #sid# and #hash# which should be replaced before you call the callback URLs.

    Replace #sid# by trackerId which was passed to you when Research Now redirected user to your survey.

    Replace #hash# with client-side generate hash.
    #hash# = MD5 of (status + trackerId + shared secret_key)

    See secure exit redirects for an example

    • Show child attributes

    • Complete
      string, URL encoded

      Redirect URL where you should send respondents to after they complete the survey successfully.

    • OverQuotas
      string, URL encoded

      Redirect URL where you should send respondents to if quota was already filled.

    • ScreenOut
      string, URL encoded

      Redirect URL where you should send respondents to if they disqualified from the survey based on screening questions.

  • resultCode
    0 or 1

    1 for success, 0 for error

  • resultError
    string,
    only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Update Project Status

POST /projects/:{extProjectId}/{pause | launch | close}

This method updates a project status with specified action.

Project status also changes the status at the quota level. To ensure that it does not changes all the quotas, pass a list of Quota Ids as mentioned in the example request

Action

  • pause

Targeting Attributes and Nesting

Targeting allows you to obtain different number of completes from different types of end users by associating a percentage of the required completes desired with an attribute option.

The most simplest form of targeting is to provide all the targeting attributes id(s) within a single nested parameter. For example, you can specify a requirement like:

  • 100% of respondents to be male or female, and
  • 100% of respondents to be within the age group of 18-25 or 26-45

In this above scenario it is likely that all males could be of the age group 18-25 and all females within the age group of 26-45, or any combination between the two.

If you desire specific control over the mix of attributes within a composition then you may choose to define nested parameter. In the above scenario with nesting you can achieve a mix to be like:

  • 25% male within the age group of 18-25,
  • 25% male within the age group of 26-45,
  • 25% female within the age group of 18-25, and
  • 25% female within the age group of 26-45.
Nesting allows you to have a fine grain control over the mix of attributes.
Example with one nest
{
  "attributes": {
    "sex": [
      {
        "option": "1", 
        "perc": "50"
      },
      {
        "option": "2", 
        "perc": "50"
      }
    ],
    "age": [
      {
        "option": "18-25", 
        "perc": "20"
      },
      {
        "option": "26-45", 
        "perc": "80"
      }
    ]
  },
  "nested": [
    [
      "age"
    ],
    [
      "sex"
    ]
  ]
}

The above example will target the following panel composition:

  • 50% of the respondents will be male and the other 50% female.
  • 20% of the respondents will be within the age group 18-25 and the other 80% within the age 26-45.
  • Since age group is not nesting with sex, it is possible that you may get a panel composition as:
    • 20% male within the age of 18-25,
    • 30% male within the age of 26-45, and
    • 50% female within the age of 26-45.

 

Example with multiple nest
{
  "attributes": {
    "sex": [
      {
        "option": "1", 
        "perc": "50"
      },
      {
        "option": "2", 
        "perc": "50"
      }
    ],
    "age": [
      {
        "option": "18-25", 
        "perc": "50"
      },
      {
        "option": "26-45", 
        "perc": "50"
      }
    ],
    "q121": [
      {
        "option": "qx121-2", 
        "perc": "50"
      },
      {
        "option": "qx121-1", 
        "perc": "50"
      }
    ]
  },
  "nested": [
    [
      "age", 
      "sex"
    ],
    [
      "q121"
    ]
  ]
}

The above example will targets the following panel composition:

  • 25% male within age 18-25
  • 25% male within age 26-45
  • 25% female within age 18-25
  • 25% female within age 26-45
  • Of the above composition 50% will be single and 50% will be married

Secure Exit Redirects

After the user completes the survey, user should be redirected to a secure exit redirect links. The link contains an encrypted hash which Research Now validates before updating the user status for the session.

The hash parameter can be generated as follows:
#hash# = MD5(status + trackerId + secret_key)

where,
status = C for completes, P for screenout and F for overquota;
trackerId = Research Now passed user’s session identifier along with the survey entry link;
secret_key = A security key provided to you by Research Now.

Please do not expose your key with anyone or in public. This could compromise the integrity of your data.
Example to compute hash

To compute hash, let’s say you have:

  • status = C
  • trackerId = 123a456
  • secret_key = The dog jumped over the house

then,

#hash# = md5(C123a456The dog jumped over the house)

#hash# = 7a0ee4af327cfee9854d963e2a2b1fbf

The resulting redirect URL becomes:

https://api-callbacks.peanutlabs.com/transactionComplete.php?trackerId=123a456&status=C&h=7a0ee4af327cfee9854d963e2a2b1fbf

Using Prescreeners

Prescreeners are a way to ask a few questions to qualify a user for a survey, without actually taking them to the client survey. Prescreeners should only be used to prescreen on attributes that we don't already know about the users. To see the list of attributes we support targeting on, check List of Available Attributes

  • title
    string

    Survey title.

  • first
    string, only alphanumerics and hyphens ‘-’ allowed

    Id for the first question.

  • head
    string

    A description that an user sees at the start of the prescreener.

  • requirements
    string

    Requirements for the survey which is displayed after prescreener has been completed.

  • question
    array of hash

    List of all prescreener questions.

    • Show child attributes

    • id
      string, only alphanumerics and hyphens ‘-’ allowed

      An unique prescreener question Id.

    • type
      integer

      Type of a question as defined by the numbers below:
      Radio button: 1
      Select/dropdown: 2
      Checkboxes: 3
      Message only: 9
      ‘Message only’ type does not have any controls. It can be used to display any message in the prescreener.

    • next
      string, alphanumeric and hyphen (-) characters allowed

      Id of the question that should follow after this question.

    • header
      string

      Placeholder for question title.

      • Show child attributes

      • html
        0 or 1

        1 indicates HTML content; 0 indicates plain text.

      • content
        string

      • Show child attributes

        array

          string

          0 or 1
