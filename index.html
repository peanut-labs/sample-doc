Sample API

Overview

Peanut Labs Sample API, a REST based API, can be programmatically used to connect your client surveys directly with Peanut Labs panelists. This API document describes the usage and resources needed for integrating with Sample API.

API Structure

Sample API is organized around REST. Below table describes the Sample API URI structure:

  • URI

    {Base URI}/{Version}/{Resources}

  • Base URI

    sample.peanutlabs.com/sample (Production)
    api-dev.peanutlabs.com/sample (Test)

  • /Version

    v1

  • /Resources

    • /projects [GET | POST]
    • /quotas [GET | POST]
    • /feasibility [POST]

  • Response.ContentType

    application/JSON

Getting Started

Authentication & Authorization

You authenticate all your requests with Sample API server by sending an HTTP authorization header as specified below:

advertiser’sId:timestamp:HMAC

Sample API server verifies the HTTP Authorization Header before allowing access to Peanut Labs system.

Here are all the parameters you will need to generate an HTTP Authorization Header.

  • Advertiser’s ID

    A unique identifier provided to you by Peanut Labs.

  • Security Key

    A secret key provided to you by Peanut Labs.

    Please do not expose your key with anyone or in public. This could compromise the integrity of your data.

  • timestamp

    Timestamp when the request was generated.

  • HMAC

    Encrypted SHA256 hash to verify your request.
    HMAC = SHA256 of (request_method + request_uri + all API parameters in alphabetical order + security_key + timestamp)

Example

Let’s say we have the following:

  • request_method = POST
  • request_uri = /sample/v1/projects
  • params = {“title”:”Display Title”, “link”:”www.google.com”, “attributes”:[“male”,”female”]}
  • request timestamp = 1433895246
  • secret_key = The dog jumped over the house

then, the parameter for HMAC function ends up as:
SHA256(POST/sample/vi/projects[“male”,”female”]www.google.comDisplay Title1433895246The dog jumped over the house)

which results in:
#hash# = fsadihtnch14hoi31

The above #hash# should be passed as HTTP Authorization Header with all HTTP requests.
HTTP Authorization Header = 1234:1439502620:fsadihtnch14hoi31

where,
1234 is the advertiser’s Id,
1439502620 is the current timestamp,
and fsadihtnch14hoi31 is the computed #hash#

Response & Error codes

Sample API uses standard HTTP status code to indicate success or failure of an API request. In general 2xx range would indicate success, 4xx range would indicate an error (such as missing parameters, invalid field or authorization failure) and 5xx will indicate an issues with Sample API server.

Sample API response will also return an application/JSON object with more information about the API operation - a resultCode to indicate success or failure of an API operation; an optional resultError to indicate a detailed error description, if any; and an optional data field to return data from the API operation.

  • resultCode
    0 or 1

    1 for success; 0 for error.

  • resultError
    string, optional only present when resultCode = 0

    A detailed error message describing what went wrong.

  • <data>
    optional

    A placeholder to return the result of an API operation.

HTTP Status code summary

  • 200

    Everything worked as expected

  • 400

    Bad request

  • 403

    Invalid security hash

  • 404

    Resources not found

  • 500

    Something went wrong on Peanut Labs end

 

Example 1: Success response

HTTP Status Code: 200
Content: { resultCode : 0 }

Example 2: Error response

HTTP Code: 400
Content: { resultCode : 1, resultError: "Project doesn't exist" }

Example 3: Error response

HTTP Status Code: 403
Content: { resultCode : 1, resultError: “Invalid Security Hash” }

API Methods

Setup a Project

POST /projects

Creating a project is the first step to launching your surveys to Peanut Labs panelist. This API is used to create a project.

Request

  • extProjectID
    required string

    A unique identifier for your project.

  • clientName
    string

    An identifier that you can use to reference your client name.

  • title
    string, URL encoded

    A project title of your choosing.

  • link
    Required string, URL encoded

    Survey URL where Peanut Labs would send its panelist to take the survey.


    Link should contain a placeholder #sid# which Peanut Labs will replace with an unique session identifier trackerId to track respondents.

  • country
    REQUIRED string

    2 letter ISO country codes.

  • currency
    string

    USD is the only currency supported.

  • notificationEmail
    string

    The email address to receive notifications, if project was paused due to conversion rules.

  • deviceTargeting
    hash

    Placeholder for device_type list.

    • Show child attributes

    • device_type
      array of string; defaults to all devices

      Devices our respondents will take the survey on. Supported devices are desktop, mobile, and tablet.

      You don't need to provide this if your survey is available on all devices.

  • inclusions
    hash

    Inclusions will let you expose traffic to respondents who were part of your previous projects.
    See inclusion/exclusion for an example.

    • Show child attributes

    • completes
      array of extProjectId

      Includes respondents who had completed your previous projects.

    • respondents
      array of extProjectId

      Includes respondents who had attempted your previous projects.

    • dropouts
      array of extProjectId

      Includes respondents who were dropped out of your projects midway.

    • overquota
      array of extProjectId

      Includes respondents who got disqualified from previous projects because of overquota.

    • screenout
      array of extProjectId

      Includes respondents who were screened out of previous projects.

  • exlusions
    hash

    Like inclusions, exclusions can be used to NOT expose traffic to respondents who were part of your previous projects.
    See inclusion/exclusion for examples.

    • Show child attributes

    • completes
      array of extProjectId

      Excludes respondents who had completed your previous projects.

    • respondents
      array of extProjectId

      Excludes respondents who had attempted your previous projects.

    • dropouts
      array of extProjectId

      Excludes respondents who were dropped out of your projects midway.

    • overquota
      array of extProjectId

      Excludes respondents who got disqualified from previous projects because of overquota.

    • screenout
      array of extProjectId

      Excludes respondents who were screened out of previous projects.

  • preScreener
    hash prescreener; defaults to Peanut Labs prescreener to test user quality

    A set of screening question that users take before qualifying for a survey. See precreener example on how to define prescreeners.

    No prescreener will be used for invalid preScreener definition.

  • autoLaunch
    boolean; false to launch in test mode, true to launch to live traffic; defaults to false

    This controls project launch. Setting this to true will launch the project as soon as it is created. False would only launch the project in test mode allowing you to test all the redirect links.

    We highly recommend you test your projects for all the redirect links before taking it live.

  • quotaSetting
    required array of hash

    Defines selection criteria for panelists according to attributes such as age, sex, etc.

    • Show child attributes

    • extQuotaID
      required string

      A unique quota identifier across all your projects.

    • completes
      required integer >= 0

      Total number of completes required for this quota.

    • cpi
      required decimal >= 0.00

      Amount Peanut Labs will be paid for each completes in USD for this quota.

      USD is the only currency supported.

    • attributes
      required hash; each key in the hash is the attributeId of that attribute

      Targeting attributes for panelists within this quota. Total allocation within each sub-attributes should sum up to 100. See attributes and nesting for examples.

      • Show child attributes

      • option
        string

        attributeId option from the list of available targeting attributes.

      • perc
        0 <= integer <= 100

        Percentage of respondents desired for this attributeId.

    • filters
      hash of array; each key in the hash is the attributeId of that attribute

      Filters are just like attributes which all panelists should belong to before qualifying for a survey. See filters for examples.

      Zip code can only be used as filters. An API call cannot have more than 10,000 zip codes in the filter list.

    • dailyCap
      integer >= 0; 0 means no limit

    • nested
      array

      Nesting will apply an inner level targeting within an already targeted group.

      See attributes and nesting for examples.

      Nesting can run upto three level deep.

 

Response

  • callbacks
    hash

    Callback URL where you should redirect respondents after the survey. Each callback URL contains two placeholders - #sid# and #hash# which should be replaced before you call the callback URLs.

    Replace #sid# by trackerId which was passed to you when Peanut Labs redirected user to your survey.

    Replace #hash# with client-side generate hash.
    #hash# = MD5 of (status + trackerId + shared secret key)

    See secure exit redirects for an example

    • Show child attributes

    • Complete
      string, URL encoded

      Redirect URL where you should send respondents to after they complete the survey successfully.

    • OverQuotas
      string, URL encoded

      Redirect URL where you should send respondents to if quota was already filled.

    • ScreenOut
      string, URL encoded

      Redirect URL where you should send respondents to if they disqualified from the survey based on screening questions.

  • resultCode
    0 or 1

    1 for success, 0 for error

  • resultError
    string,
    only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

    REQUIRED string

    A unique quota Id.

    Quota Id should be unique across all your projects

  • completes
    REQUIRED integer >= 0

    Total number of completes required for this quota.

  • cpi
    REQUIRED decimal >= 0.00

    Amount Peanut Labs will be paid for each completes in USD for this quota.

    USD is the only currency supported.

  • attributes
    REQUIRED hash; each key in the hash is the attributeId of that attribute

    Targeting attributes for panelists within this quota. Total allocation within each sub-attributes should sum up to 100. See attributes and nesting for examples.

    • Show child attributes

    • option
      string

      attributeId option from the list of available targeting attributes.

    • perc
      0 <= integer <= 100
      string

      Percentage of respondents desired for this attribute.

  • filters
    hash of array; each key in the hash is the attributeId of that attribute

    Filters are just like attributes which all panelists should belong to before qualifying for a survey. See filters for examples.

    Zip code can only be used as filters. An API call cannot have more than 10,000 zip codes in the filter list.

  • dailyCap
    integer >= 0; 0 means no limit

    Maximum number of daily completes desired for this quota.

  • nested
    array

    Nesting will apply an inner level targeting within an already targeted group.

    See attributes and nesting for examples.

  • autoLaunch

Get Project Test Link

GET /projects/:{extProjectID}/test

Test link API is only called when project autoLaunch is set to false. When you create a project with autoLaunch set to false, you can use our test publishers to test your project workflow before taking your project live.

This method will never be used to once you have integrated your system with Sample API.

Request

  • screener
    required boolean; false to disable, true to enable; defaults to true

    When true the response will contain a test link to Peanut Lab Reward Center.
    When false the response will contain a direct test link to the survey.

 

Request

  • link
    string

    Test link to Peanut Lab Reward Center or to the survey.

  • resultCode

  • resultError string, only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Definition

}

Create Feasibility Count

POST /feasibility

Estimates the feasibility of a project based required attributes. Response will return a possible quota group (based on input attributes) and an estimated number of completes possible within each quota group.

Request

  • attributes
    required hash; each key in the hash is the attributeId of that attribute

    Targeting attributes for panelists within this quota. Total allocation within each sub-attributes should sum up to 100. See attributes and nesting for examples.

    • Show child attributes

    • option
      string

      attributeId option from the list of available targeting attributes.

    • perc
      0 <= integer <= 100

      Percentage of respondents desired for this attribute.

  • filters hash of array; each key in the hash is the attributeId of that attribute

    Filters are just like attributes which all panelists should belong to before qualifying for a survey. See filters for examples.

    Zip code can only be used as filters. An API call cannot have more than 10,000 zip codes in the filter list.

  • days integer >= 0

    Number of days feasibility estimates is based on. This is the days in field for the project.

 

Response

  • data
    array

    Place holder for response data

    • Show child attributes

    • node
      hash

      Attribute Id(s) and attribute values(s) from targeting attributes list.

    • count
      integer

      Number of possible completes for this node.

  • resultCode 0 or 1

    1 for success; 0 for error.

  • resultError string,
    only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Definition

POST

http://api-dev.peanutlabs.com/sample/v1/feasibility

   "age":[  

     "22-60"

   ],

     "US"

   ]

 },

 "days":"5"

}

 

Example Response

{

   {  

     "node":{  

       "age":"22-60",

       "cc":"US",

     "node":{

Update Project Status

POST /projects/:{extProjectId}/{pause | resume | launch | close}

This method updates a project status with specified action.

Action

  • pause

    Stop traffic to all quotas for this project.

  • resume

    Resume traffic to all quotas for this project.

  • launch

    Takes a project live for the first time. This is only used when project was created with autoLaunch set as false.

  • close

    Stops all traffic flow into the project and closes it. Once a project is closed it cannot be reopened.

    Only use this when you have determined your project is complete and can be closed.

 

Response

  • resultCode
    0 or 1

    1 for success; 0 for error.

  • resultError
    string, only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Definition

POST

http://api-dev.peanutlabs.com/sample/v1/projects/project001/pause

Example Response

{  

 "resultCode":"1"

}

Update Quota Status

POST /quotas/:{extQuotaId}/{pause | resume | launch | close}

This method updates a quota status. Following are the four possible operations on a quota:

Action

  • pause

    Stops traffic to a quota.

  • resume

    Resume traffic to a quota.

  • launch

    Takes a quota live for the first time. This is only used when quotaSetting was created with autoLaunch is set to false.

    If the project was not already launched, launching a quota will also launches the project to live traffic with this quota.

  • close

    Stops all traffic flow into a quota and close it completely. Once a quota is closed it cannot be reopened.

    Only use this when you have determined your project quota is complete and can be closed.

 

Response

  • resultCode
    0 or 1

    1 for success; 0 for error.

  • resultError
    string, only present if resultcode = 0

    Descriptive error message. See response and error codes for details.

Definition

POST

http://api-dev.peanutlabs.com/sample/v1/quotas/invite001/pause

Example Response

 "resultCode":"1"

}

Targeting Attributes and Nesting

Targeting allows you to obtain different number of completes from different types of end users by associating a percentage of the required completes desired with an attribute value.

The most simplest form of targeting is to just provide the targeting attributes without any nesting structure. With this targeting, you can specify a requirement like:

  • 100% of respondents to be male or female, and
  • 100% of respondents to be within the age group of 18-25 or 26-45
  • without caring to filter within within each attributes.

In this above scenario it is likely that all males could be of the age group 18-25 and all females within the age group of 26-45, or any combination between the two.

If you desire specific control over the mix of attributes within a composition then you may choose to define nested attributes. In the above scenario with nesting you can achieve a mix to be like:

  • 25% male within the age group of 18-25,
  • 25% male within the age group of 26-45,
  • 25% female within the age group of 18-25, and
  • 25% female within the age group of 26-45.
Nesting allows you to have a fine grain control over the mix of attributes.
Example without nesting

 "attributes":{  

   "sex":[  

     {  

       "option":"1",

       "perc":"50"

     },

     {  

       "option":"2",

       "perc":"50"

     }

   ],

   "age":[  

     {  

       "option":"18-25",

       "perc":"20"

     },

     {  

       "option":"26-45",

       "perc":"80"

     }

   ]

 }

 

The above example will target the following panel composition:

  • 50% of the respondents will be male and the other 50% female.
  • 20% of the respondents will be within the age group 18-25 and the other 80% within the age 26-45.
  • Since age group is not nesting with sex, it is possible that you may get a panel composition as:
    • 20% male within the age of 18-25,
    • 30% male within the age of 26-45, and
    • 50% female within the age of 26-45.
Example with nesting

 

 "attributes":{  

   "sex":[  

     {  

       "option":"1",

       "perc":"50"

     },

     {  

       "option":"2",

       "perc":"50"

     }

   ],

   "age":[  

     {  

       "option":"18-25",

       "perc":"50"

     },

     {  

       "option":"26-45",

       "perc":"50"

     }

   ],

   "q121":[  

     {  

       "option":"qx121-0",

       "perc":"50"

     },

     {  

       "option":"qx121-1",

       "perc":"50"

     }

   ]

 },

 "nested":[  

   [  

     "age",

     "sex"

   ],

   [  

     "q121"

   ]

 ]

The above example will targets the following panel composition:

  • 25% male within age 18-25
  • 25% male within age 26-45
  • 25% female within age 18-25
  • 25% female within age 26-45
  • Of all the above composition, there will be 50% African American and 50% Caucasian

Filters

Filters are attributes where all users should have before qualifying for the survey.

  • Zip code can only be used as filters
  • An API call cannot have more than 10,000 zip codes in the filter list

{"q101":["qx101-1","qx101-2","qx101-3"]}

Respondents without attribute qx101-1, qx101-2, qx101-3 will not be invited for the survery.

Inclusions/Exclusions

Projects can include/exclude certain target based on completes, respondents, dropouts, screenouts and overquotas. Inclusions/exclusions are filtered by extProjectId.

Example 1

{"completes": ["129","130"], "respondents": ["150","156"]}

  • include/exclude those who have completed projects with extProjectId 129 or 130
  • include/exclude those who have responded to a project with extProjectId 150 or 156
Example 2

{"completes": ["129","130"], "respondents": ["150","156"], "dropouts": ["131"], "overquota": ["132"], "screenout": ["133"]}

  • include/exclude those who have completed projects with extProjectId 129 or 130
  • include/exclude those who have responded to projects with extProjectId 150 or 156
  • include/exclude those who have dropped out of project with expProjectId 131
  • include/exclude those who have been over quota of project with extProjectId 132
  • include/exclude those who have been screened out of project with extProjectId 133

Secure Exit Redirects

After the user completes the survey, user should be redirected to a secure exit redirect links. The link contains an encrypted hash which Peanut Labs validates before updating the user status for the session.

The #hash# parameter can be generated as follows:
#hash# = MD5(status + trackerId + Security Key)

where,
status = C for completes, P for screenout and F for overquota;
trackerId = Peanut Labs passes user session identifier with the survey;
Security Key = A secret key provided to you by Peanut Labs.

Please do not expose your key with anyone or in public. This could compromise the integrity of your data.

When a user completes a survey, the exit redirect used is:
http:\/\/api-callbacks.peanutlabs.com\/transactionComplete.php?trackerId=#sid#&status=#status#&h=#hash#

where,
#status# = C for completes, P for screenouts and F for overquotas

To compute #hash#, let’s say you have:

  • status = C
  • trackerId = 123a456
  • security key = 347602146a923872538f3803eb5f3cef

then,
md5(C123a456347602146a923872538f3803eb5f3cef) of results in
h = aed04fb1182a42cf35fe061b33f82fee

Hence, the resulting redirect URL will be:
http:\/\/api-callbacks.peanutlabs.com\/transactionComplete.php?trackerId=123a456&status=C&h=aed04fb1182a42cf35fe061b33f82fee

Prescreener Format

  • title
    string

    Survey title.

  • first
    string, only alphanumerics and hyphens ‘-’ allowed

    Id for the first question.

  • head
    string

    A description that an user sees at the start of the prescreener.

  • requirements
    string

    Requirements for the survey which is displayed after prescreener has been completed.

  • question
    array of hash

    List of all prescreener questions.

    • Show child attributes

    • id
      string, only alphanumerics and hyphens ‘-’ allowed

      An unique prescreener question Id.

    • type
      integer

      Type of a question as defined by the numbers below:
      Radio button: 1
      Select/dropdown: 2
      Checkboxes: 3
      Open input text: 4
      Date: 5
      Gred: 15
      Message only: 15
      ‘Message only’ type does not have any controls. It can be used to display any message in the prescreener.

    • next
      string, alphanumeric and hyphen (-) characters allowed

      Id of the question that should follow after this question.

    • header
      string

      Placeholder for question title.

      • Show child attributes

      • html
        boolean

        true indicates HTML content; false indicates plain text.

      • content
        string

        Actual question that will be displayed to an user.

    • cellHeaders
      hash

      Placeholder for grid question headers.

      Only applicable for grid questions.
      • Show child attributes

      • option
        array

        Placeholders for grid question headers.

        • Show child attributes

        • id
          string, alphanumeric and hyphen (-) characters allowed

          An unique cell header Id.

        • title

          Cell header title

    • options
      hash

      Placeholder for multiple answer option.

      • Show child attributes

      • option
        array

        Placeholder for multiple answer option.

        • Show child attributes

        • id
          string, alphanumeric and hyphen (-) characters allowed

          An unique Id among the option set.

        • title
          string

          Text to display for an option.

        • value
          string

          Specifies the value which can be used to apply sort. Not used otherwise.

        • target
          string

          Question id or workflow to branch to. This option is overrides the next of question defined earlier. Allowed values for workflow is complete or screenout.

        • exclusive
          boolean

          Makes an option exclusive i.e. if this option is selected, then no other option can be selected.

        • cells
          hash

          Placeholder for cells.

          Only applicable for grid questions.
          • Show child attributes

          • id
            string, alphanumeric and hyphen (-) characters allowed

            An unique Id among the option set.

          • value
            string

            Specifies the value which can be used to apply sort. Not used otherwise.

          • checked
            boolean

            true will select the cell; false will not select the cell.

    • option-sort-field
      string

      Determines how options are sorted. Can only be id or value

    • option-sort-order
      string

      Can only be desc, asc or rand. An option with exclusive as true will not be sorted and will always appear at the end.

    • skip-logic
      array

      Set of skip-logic that will be executed based on user selected options.

      • Show child attributes

      • options
        array

        List of Id from the options attribute.

      • target
        string

        Question id or workflow to branch to. This option is overrides the next of question defined earlier. Allowed values for workflow is complete or screenout.

      • priority
        integer

        skip-logic is sorted based on descending order of priority. skip-logic with highest priority will be executed first.

    • required
      hash

      Presence of required attribute makes the question response mandatory.

      • Show child attributes

      • message
        string

        Message to display when user tries to advance without answering the mandatory question.

    • ranges
      hash

      Placeholder for date ranges.

      Only applicable for date type question.
      • Show child attributes

      • range
        array

        Placeholder for a date range.

        • Show child attributes

        • min
          long integer

          Unix timestamp for minimum date range.

        • max
          long integer

          Unix timestamp for maximum date range.

        • target
          string

          Branch to question id or workflow, if value is in the range. Allowed values for workflow is complete or screenout.
